# AZABU+ Project 次期展開設計書

最終更新: 2026-04-03
対象: GitHub Pages 運用中の AZABU+ Project

## 1. 目的

本設計書は、現状の施策（ブランドハブ、記事群、CTA、計測の一部実装）を前提に、次の3-6ヶ月で「集客」「信頼」「CV」を安定成長させるための実行計画を定義する。

## 2. 目標（KGI/KPI）

- KGI1: 面談予約数の増加（前四半期比で継続増）
- KGI2: ブランド検索流入の増加（`AZABU+転職` 系）
- KGI3: 非ブランド流入からCVまでの到達率向上

主要KPI（週次確認）:
- セッション数（全体 / 自然検索）
- 記事別ランディング数（上位20本）
- ブランドハブ到達数（`/azabu-plus-tenshoku`）
- `cv_click` イベント数（`button_location` 別）
- 記事 → ブランドハブ → 予約 の遷移率

## 3. 施策方針

### 3.1 集客
- AIO/SEOを前提に、悩み起点（空白期間、学歴不安、スキル不足）で記事を拡張。
- 既存100記事は「新規追加」より「改善優先」を導入（CTR・滞在時間・内部遷移の改善）。

### 3.2 信頼
- FAQ、構造化データ、ブランド表記ゆれ統合を継続。
- 役員と直接話せる価値、未経験支援の実績/具体性を全導線で一貫表示。

### 3.3 CV
- CTA文言・配置を統一し、クリック計測を漏れなく実装。
- 高流入記事にCV近接ブロック（不安解消 + 面談予約）を優先配置。

## 4. フェーズ設計

## Phase A（2-4週間）: 計測基盤の完成

目的:
- どの導線が予約に効いているかを、定量で判定できる状態にする。

実施内容:
- `cv_click` の統一運用（event名固定、`button_location` 標準化）
- 全カレンダーリンクの計測実装確認
- GA4で主要イベントをコンバージョン登録
- 週次レポートフォーマット作成

完了条件:
- 主要CTAの計測漏れゼロ
- 週次で `button_location` 比較が可能

## Phase B（4-8週間）: コンテンツ運用の型化

目的:
- 記事量産から、記事改善サイクルへ移行する。

実施内容:
- 記事を3分類で管理
  - 集客記事（検索入口）
  - ハブ誘導記事（ブランド理解）
  - CV直前記事（不安解消）
- 新規/リライトの比率を定義（例: 新規3 : 改善7）
- 内部リンクの必須ルール化（記事→ハブ→予約）

完了条件:
- 各記事の役割が台帳化されている
- 優先20記事の改善が完了

## Phase C（4-10週間）: 技術SEOとUX最適化

目的:
- 検索評価の安定化と離脱率低下。

実施内容:
- `sitemap.xml` / `robots.txt` / canonical の整合チェック
- Core Web Vitals 最適化（画像、JS分割、不要処理削減）
- 構造化データの追加/整備（FAQ, Article, Organization）

完了条件:
- 技術SEOチェックリストの全項目クリア
- モバイル体験指標の改善傾向

## Phase D（中期）: 運用拡張の意思決定

目的:
- 編集体制に応じて CMS 導入可否を判断。

判断基準:
- 月次更新本数
- 編集者人数
- 非エンジニア編集ニーズ
- 反映速度要件

推奨:
- まず GitHub Pages + 現行運用で成果最大化
- 運用負荷が閾値を超えた段階で Headless CMS 検討

## 5. 実行体制

- 企画/編集: 記事テーマ選定、リライト指示
- 実装: 計測、CTA、内部リンク、SEO技術項目
- 分析: 週次レポート、改善案提示、優先順位更新

運用会議（週1回）議題:
- KPI差分
- 上位流入記事の改善優先順位
- 次週の実装タスク確定

## 6. 優先タスク（直近2週間）

1. カレンダー導線の計測網羅確認（未実装有無の棚卸し）
2. `button_location` 命名規約の確定（ドキュメント化）
3. 高流入記事10本のCTA改善（文言・配置・内部リンク）
4. ブランドハブの比較テスト案作成（ヒーローCTA周辺）
5. 週次ダッシュボードのテンプレ運用開始

## 7. 命名規約（推奨）

`cv_click` 送信用の `button_location` は以下のように命名する:
- `home_page_cta`
- `mobile_sticky_cta`
- `mobile_menu_cta`
- `brand_hub_hero_cta`
- `article_brand_hub_cta`

規約:
- 小文字スネークケース
- ページ_セクション_要素
- 意味が重複する既存名は統合

## 8. リスクと対策

- リスク: 記事数増加による品質ばらつき
  - 対策: 新規より改善重視、品質チェック項目固定
- リスク: 計測の欠損/重複
  - 対策: 共通関数化、命名規約、QAチェック導入
- リスク: 早すぎるCMS導入で複雑化
  - 対策: 定量条件を満たすまで現行運用を継続

## 9. 成果判定（四半期）

以下を満たせば次フェーズへ進む:
- 主要KPIが前月比で改善傾向
- 計測欠損が解消
- 優先記事群でCV寄与が確認できる
- 運用体制が安定し、改善サイクルが回っている

---

本設計書は運用しながら更新する。まずは Phase A を完遂し、計測データに基づいて Phase B/C の優先順位を毎週再調整する。
